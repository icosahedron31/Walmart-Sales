# Walmart Sales Forecasting — README

## შესავალი

ეს დოკუმენტი აღწერს Walmart Recruiting - Store Sales Forecasting-ის ამოცანაზე ჩატარებულ ექსპერიმენტებს, თითოეული მოდელის არქიტექტურისთვის ცალ-ცალკე. თითოეულ სექციაში აღწერილია: მონაცემთა მომზადება, ტესტირებული მიდგომები, აღმოჩენილი პრობლემები და მათი გადაწყვეტა, და საბოლოო შედეგები.

## LightGBM

### მოდელის მიმოხილვა

LightGBM გამოყენებულია როგორც tree-based baseline მოდელი, XGBoost-ის პარალელურად. ორივე მოდელი იყენებს იდენტურ feature set-სა და იდენტურ train/val split-ს (cutoff date: 2012-04-01), რაც პირდაპირ შედარების საშუალებას იძლევა.

**გამოყენებული features (27):**
`Store, Dept, Week, Month, Year, Quarter, IsHoliday, holiday_name, Type, Size, Temperature, Fuel_Price, CPI, Unemployment, MarkDown1-5, lag_1, lag_2, lag_4, lag_52, rolling_mean_4, rolling_mean_12, rolling_std_4, rolling_mean_52`

`Type` და `holiday_name` categorical სვეტებია, დამუშავებული `CategoryConverter` custom transformer-ის საშუალებით, pipeline-ის შიგნით.

LightGBM-ის დამუშავება რამდენიმე ეტაპად წარიმართა: (1) საწყისი ტესტირება obj. ფუნქციისა და ჰიპერპარამეტრების გარშემო, (2) დამატებითი holiday proximity features-ის ექსპერიმენტი, და (3) საბოლოო, დახვეწილი ვერსია სრული მეტრიკების ლოგირებით. თითოეული აღწერილია ქვემოთ.

### 1. მონაცემთა წმენდა (`LightGBM_Cleaning`)

დამუშავებამდე მოშორებულია სტრიქონები, სადაც `lag_52` ან `rolling_mean_52` არის NaN (ანუ სერიები, რომლებსაც 52 კვირაზე ნაკლები ისტორია აქვთ). დარჩენილია 173,973 სტრიქონი train-ში.

### 2. საწყისი ექსპერიმენტები: Objective Function (L2 vs L1) და მანუალური ტუნინგი

პირველ ეტაპზე შემოწმდა, თუ რამდენად მნიშვნელოვანია loss function-ის არჩევანი. LightGBM-ის default objective — `regression` (L2/MSE) — კვადრატულ ცდომილებას ხდის მინიმალურს, მაშინ როცა WMAE აბსოლუტურ ცდომილებაზეა დაფუძნებული. ამიტომ შემოწმდა `regression_l1` (MAE) obj.-ის გავლენაც.

| კონფიგურაცია | Objective | lr | leaves | depth | WMAE |
|---|---|---|---|---|---|
| Baseline (100 trees) | L2 | 0.1 | default | default | 1423.58 |
| V1 | L2 | 0.05 | 63 | 8 | 1375.37 |
| V2 | L2 | 0.01 | 31 | 6 | 1391.12 |
| Grid Search | L2 | 0.05 | 63 | 7 | 1372.24 |
| Baseline (100 trees) | L1 | 0.1 | default | default | 1431.76 |
| **V1** | **L1** | **0.05** | **63** | **8** | **1361.80 ← საუკეთესო** |
| V2 | L1 | 0.01 | 31 | 6 | 1363.03 |
| Grid Search | L1 | 0.05 | 31 | 10 | 1363.44 |

**აღმოჩენები:**
- **L1 objective-ზე გადასვლამ პირდაპირ გააუმჯობესა შედეგი** — V1 კონფიგურაციაზე 1375.37-დან 1361.80-მდე (13.57 ერთეულით), ლოგიკურად, რადგან L1 (MAE) უფრო ახლოს დგას WMAE-სთან, ვიდრე L2 (MSE).
- L1-ზე გადასვლისას მოდელს მეტი ხე დასჭირდა კონვერგენციისთვის (674 iteration L2-ის 105-ის წინააღმდეგ) — მოსალოდნელია, რადგან L1 obj. უფრო რთულია ოპტიმიზაციისთვის.
- **Feature importance-ის რანჟირება შეიცვალა obj.-ის მიხედვით:** L2-ზე `lag_52` იყო #1, ხოლო L1-ზე `Dept` გახდა #1 და `lag_52` დაეცა #3-ზე — მოდელმა სხვანაირად გადაანაწილა ყურადღება ცდომილების ტიპის მიხედვით.
- Grid search ამ ეტაპზე **ვერ გაუსწრო მანუალურად ნაპოვნ საუკეთესო კონფიგურაციას** (1363.44 vs 1361.80) — ავტომატური ძებნა ყოველთვის არ პოულობს გლობალურ ოპტიმუმს ვიწრო grid-ის პირობებში.

**საბოლოო არჩევანი ამ ეტაპისთვის:** `objective='regression_l1', learning_rate=0.05, num_leaves=63, max_depth=8`, WMAE = **1361.80**.

### 3. დამატებითი ექსპერიმენტი: Holiday Proximity Features

შემდეგ ეტაპზე შემოწმდა, აუმჯობესებს თუ არა შედეგს დამატებითი, უფრო დახვეწილი holiday-related features. Preprocessing pipeline-ს დაემატა `HolidayProximityTransformer` — custom transformer, რომელიც ითვლის `weeks_to_{holiday}`, `weeks_since_{holiday}` და ციკლურ `{holiday}_phase_sin`/`{holiday}_phase_cos` მახასიათებლებს ოთხივე ძირითადი დღესასწაულისთვის (Super Bowl, Labor Day, Thanksgiving, Christmas) — სულ 16 დამატებითი სვეტი.

**Feature importance ახალი სვეტებით:**
- ახალი holiday features (`labor_day_phase_sin`, `weeks_to_super_bowl`, `weeks_to_thanksgiving`, thanksgiving/christmas phase features) გამოჩნდნენ non-zero importance-ით
- ყველა `weeks_since_*` სვეტს ჰქონდა **ნულოვანი importance**
- `Dept` და `Store` კვლავ დომინირებდნენ
- `holiday_name`-ის importance შესამჩნევად დაეცა — ახალმა proximity features-მა უფრო ზუსტად აღწერა იგივე ინფორმაცია

**შედეგები:**

| ეტაპი | WMAE | შედარება baseline-თან |
|---|---|---|
| Baseline (100 trees, L1) | 1446.84 | უარესი (1431.76 vs) |
| V1 კონფიგურაცია | 1369.72 | უარესი (1361.80 vs) |
| Grid Search | 1365.43 | უარესი |
| `weeks_since_*` მოშორების შემდეგ | 1365.43 | უცვლელი |
| Best Params Training (extended) | 1365.43 | უცვლელი |
| **CV Mean WMAE** | **1512.92 ± 309.94** | — |

**დასკვნა: `HolidayProximityTransformer`-ის დამატებამ ვერ გააუმჯობესა LightGBM-ის შედეგი** — ვერცერთმა კონფიგურაციამ ვერ გადააჭარბა თავდაპირველ საუკეთესო შედეგს (1361.80). თუნდაც `weeks_since_*` სვეტების მოშორების შემდეგაც (რომლებსაც ნულოვანი importance ჰქონდათ), შედეგი უცვლელი დარჩა.

**რატომ ვერ დაეხმარა დამატებითი features:**
- `weeks_since_*` სვეტები მთლიანად noise იყო, სიგნალის გარეშე
- ციკლურმა sin/cos features-მა დაამატა 8 ახალი განზომილება და მოდელს მოუწია "დაესწავლა მათი უგულებელყოფა"
- მეტი feature ნიშნავს მეტ საჭირო split-ს, ხეები უფრო სწრაფად აწყდება სირთულის ლიმიტს ("No further splits with positive gain" გაფრთხილებები ამის დასტურია)
- `lag_52` და `holiday_name` უკვე საკმარისად კარგად ატარებდნენ სეზონურ/დღესასწაულურ სიგნალს — დამატებითი ინჟინერია ამ კონკრეტული მოდელისთვის ზედმეტი აღმოჩნდა

ამის შემდეგ, `HolidayProximityTransformer` მოშორდა preprocessing pipeline-ს მთლიანად, და დარჩა თავდაპირველი 27-feature ნაკრები, `holiday_name` კატეგორიული სვეტით.

#### 4.5 საბოლოო Pipeline და რეგისტრაცია (`LightGBM_Pipeline_Final`)

საბოლოო მოდელი გაწვრთნილია სრულ `X_train`-ზე, `n_estimators=1393`-ით (extended training-იდან ნაპოვნი კონვერგენციის წერტილი):


**დასკვნა:** Recursive inference, თუმცა თეორიულად მიდრეკილია compounding error-ისკენ, პრაქტიკაში **უფრო მდგრადი** აღმოჩნდა, ვიდრე მოსალოდნელი იყო — რადგან ის ინარჩუნებს ყველა test სტრიქონს training-სთვის ცნობილი განაწილების ფარგლებში (`lag_52` ყოველთვის რეალურ ისტორიაზეა დაფუძნებული). მოკლე lag features-ის მოცილებამ ეს "usable range" შეამცირა და გამოააშკარავა ახალი, უფრო სერიოზული failure mode. **`XGBoost_Best` (რეკურსიული inference, sample_weight-ის გარეშე) რჩება საბოლოო არჩევანი.**

## DLinear
 
### მოდელის მიმოხილვა და ფუნდამენტური შეზღუდვა
 
DLinear (Decomposition Linear) წარმოადგენს Deep Learning არქიტექტურას, რომელიც შედგება ორი linear layerისაგან — ერთი ტრენდისთვის, ერთი სეზონურობისთვის — `SeriesDecomp` მოდულის საშუალებით. გამოყენებულია `neuralforecast` ბიბლიოთეკის იმპლემენტაცია.
 
**ფუნდამენტური არქიტექტურული შეზღუდვა:** DLinear-ს, XGBoost-ისა და LightGBM-ისგან განსხვავებით, **არ შეუძლია exogenous ცვლადების გამოყენება**. მართალია, `neuralforecast`-ის `DLinear` კონსტრუქტორი ფორმალურად იღებს `futr_exog_list`/`hist_exog_list`/`stat_exog_list` პარამეტრებს (ყველა მოდელისთვის საერთო interface-ის გამო), მაგრამ კლასის დონეზე მითითებულია:
 
```python
class DLinear(BaseWindows):
    EXOGENOUS_FUTR = False
    EXOGENOUS_HIST = False
    EXOGENOUS_STAT = False
```
 
ანუ, არქიტექტურას თავად არ გააჩნია მექანიზმი, რომ ეს ცვლადები რეალურად გამოიყენოს prediction-ის დროს. ეს ნიშნავს, რომ DLinear ვერასდროს ისარგებლებდა `holiday_name`-ით, `MarkDown`-ებით, ეკონომიკური ინდიკატორებით და ა.შ. — მას შეუძლია ისწავლოს მხოლოდ სერიის საკუთარი წარსული მნიშვნელობებიდან (`y`), `input_size`-ის ფარგლებში.
 
**ეს ფუნდამენტურად განსხვავებული და ნაკლებად ინფორმირებული მიდგომაა** tree-based მოდელებთან შედარებით, და დიდწილად ხსნის, თუ რატომ ჩამორჩება DLinear XGBoost/LightGBM-ს ამ ამოცანაზე — არა იმიტომ, რომ არქიტექტურა "სუსტია", არამედ იმიტომ, რომ მას საერთოდ არ აქვს წვდომა იმ ინფორმაციაზე (დღესასწაულები, ფასდაკლებები, ეკონომიკური მაჩვენებლები), რომელსაც tree-based მოდელები აქტიურად იყენებენ.
 
### თარიღების შეუსაბამობის ფიქსი
 
`neuralforecast` ბიბლიოთეკა პროგნოზებს კვირის პირველ დღეს (კვირა) აბრუნებს, ხოლო Walmart-ის მონაცემები პარასკევს დათარიღებულია. შესაბამისად, merge-ამდე forecast თარიღები გადავიყვანეთ უახლოეს პარასკევზე:
 
```python
forecasts['ds'] = forecasts['ds'] - pd.to_timedelta(forecasts['ds'].dt.dayofweek - 4, unit='d')
forecasts.loc[forecasts['ds'].dt.dayofweek != 4, 'ds'] = forecasts['ds'] + pd.offsets.Week(weekday=4)
```

## Prophet
 
### მოდელის მიმოხილვა
 
Prophet არის კლასიკური სტატისტიკური დროითი მწკრივის მოდელი (Meta/Facebook-ის მიერ შემუშავებული), რომელიც თითოეულ სერიას (Store×Dept კომბინაციას) ცალ-ცალკე, დამოუკიდებლად აწვდის — განსხვავებით tree-based მოდელებისგან, რომლებიც ერთ საერთო მოდელს სწავლობენ ყველა სერიისთვის ერთად. ეს ფუნდამენტური არქიტექტურული განსხვავება ნიშნავს, რომ Prophet **ვერ სესხულობს ინფორმაციას** სერიებს შორის, მაგრამ ამის სანაცვლოდ თითოეულ სერიას სპეციფიკურ, მორგებულ მოდელს სთავაზობს.
 
**მნიშვნელოვანი განსხვავება DLinear-თან:** მართალია, Prophet-იც უმეტესწილად univariate მიდგომაა (მხოლოდ `y`-ს იყენებს თითოეული სერიისთვის), მაგრამ მას **აქვს** ჩაშენებული `holidays` პარამეტრი — ერთგვარი exogenous-ის ფორმა, რომელიც საშუალებას აძლევს მოდელს, ცალკე მოახდინოს დღესასწაულების ეფექტის მოდელირება. ეს ხდის მას შუალედურ რგოლად DLinear-სა (სრულად univariate) და tree-based მოდელებს (სრული exogenous feature set) შორის.
 
### 1. მონაცემთა მომზადება და დღესასწაულების განსაზღვრა
 
გამოყენებულია იგივე ≥52-კვირიანი ფილტრი, რაც DLinear-ში. დღესასწაულები განისაზღვრა Prophet-ის საკუთარი `holidays` DataFrame-ის საშუალებით, `lower_window=-1, upper_window=1` (ანუ დღესასწაულის ეფექტი ვრცელდება მის წინა და მომდევნო კვირაზეც):
 
```python
holidays = pd.DataFrame({
    'holiday': ['superbowl', ..., 'laborday', ..., 'thanksgiving', ..., 'christmas', ...],
    'ds': pd.to_datetime([...]),
    'lower_window': -1, 'upper_window': 1
})
```
 
### 2. Seasonality Mode: Additive vs Multiplicative

 ## Ensemble: XGBoost + LightGBM
 
ვინაიდან XGBoost-სა და LightGBM-ს, მსგავსი საბოლოო WMAE-ის მიუხედავად, სხვადასხვა feature importance პროფილი ჰქონდათ , შემოწმდა, თუ მათი პროგნოზების კომბინირება აუმჯობესებს თუ არა შედეგს — რაც მიუთითებს, ერთმანეთისგან საკმარისად დამოუკიდებელ შეცდომებზე.

ორივე კომბინირებული ვერსია სჯობს ცალკეულ მოდელებს — რაც ადასტურებს, რომ ორივე მოდელი გარკვეულ, ერთმანეთისგან განსხვავებულ ინფორმაციას ატარებს, მიუხედავად თითქმის იდენტური საბოლოო WMAE-ისა.
 
### Kaggle-ის შედეგი
 
Recursive walk-forward inference გამოყენებულია **ორივე** მოდელისთვის ცალ-ცალკე (თითოეული მოდელი საკუთარ lag features-ს აშენებს საკუთარი წინა prediction-ებიდან, რათა ერთი მოდელის შეცდომამ არ გააფუჭოს მეორის input), შემდეგ კომბინირებულია ოპტიმიზირებული წონებით:
 
| Submission | Public WMAE | Private WMAE |
|---|---|---|
| **XGBoost + LightGBM Ensemble** | **2928.32** | **3067.90** |
| XGBoost_Best (მარტო) | 3039.99 | 3191.68 |
| XGBoost_ReducedLags | 12790.50 | 13080.78 |
 
**Ensemble-მა გააუმჯობესა private score ~123.8 ერთეულით (~3.9%)** — რეალური, არატრივიალური გაუმჯობესება. მნიშვნელოვანია, რომ ამჯერად validation-ისა და Kaggle-ის შედეგები **მიმართულებით ეთანხმება ერთმანეთს** (validation-ზე ~1.2% გაუმჯობესება, Kaggle-ზე ~3.9%) — განსხვავებით `XGBoost_ReducedLags`-ის შემთხვევისგან, სადაც validation-ის "გაუმჯობესება" კატასტროფულ Kaggle-ის შედეგად გადაიქცა. მიზეზი მარტივია: ეს ექსპერიმენტი იყენებს **იმავე, უკვე დადასტურებულ** recursive inference მიდგომას ორივე მოდელისთვის — არანაირი ახალი structural mismatch validation-სა და test-ს შორის.
 
### რეგისტრაცია
 
ვინაიდან ensemble არ არის ერთი, ცალკე გაწვრთნილი მოდელი, არამედ ორი დამოუკიდებელი მოდელის კომბინაცია, საბოლოო რეგისტრაციისთვის საჭირო იყო custom `mlflow.pyfunc.PythonModel` wrapper (`EnsemblePyfunc`), რომელიც შიგნით ინახავს pipeline-ს და ოპტიმიზირებულ წონებს, და `.predict()`-ის გამოძახებისას შიგნით აწარმოებს ორივე პროგნოზს და აერთიანებს მათ. ეს pipeline დარეგისტრირებულია, როგორც **`Ensemble_Best`** — ეს არის პროექტის **საბოლოო, საუკეთესო მოდელი**.

## TFT 
tft-ს უპირატესობა შემდეგ მოდელებთან არის ის, რომ მას შეუძლია ეგზოგენური ცვლადების მიღება, როგორც სტატიკურის, ასევე დროში ცვალებადის, რაც ჩვენს ამოცანასთან შესაბამისობაში მოდიოდა. ყველა მონაცემი რომელიც მოცემული გვქონდა, შეიძლება გამოგვეყენებინა prediction-ის გასაკეთებლად. ასევე განვიხილეთ რამდენიმე დამატებითი ცვლადის შემოღებაც (feature engineering) 
### Feature Engineering 
მონაცემების პერიდულობის გათვალისწინებით, და განსაკუთრებით პიკებზე/დაღმასვლებზე დაკვირვებით holiday-ებთან ახლოს, გადავწყვიტეთ თითოეული holiday-სთვის შეგვემუშავებინა 4 ცვლადი: 
 **phase_sin**
 **phase_cos**
 **weeks_since** 
 **weeks_before** 
 ეს ცვლადები აღწერდა holiday-ებამდე როგორც ციკლურ, ისევე წრფივ დამოკიდებულებებსაც. weeks_since და weeks_before-ს დამატება გამოწვეულია იმით, რომ ზოგიერთ შემთხვევაში holiday-ის წინა კვირებში ხდებოდა დაცემა/ასვლა, ზოგჯერ შემდეგში, და ორივე მიმართულებით ინფორმაციის ცოდნა გაუადვილებდა მოდელს სწავლას. 
### feature selection 
თავდაპირველად დავატრენინგეთ მარტივი tft მოდელი hyperparameter tuning-ის გარეშე. მიღებული მოდელი არ იყო საუკეთესო თუმცა, მისგან შეგვეძლო მაენკოდირებელი ცვლადების მნიშვნელობების (feature-importance) ამოღება. ამ ყველაფრის შედეგად ცვლადების მნიშვნელობები ასე გადანაწილდა:

### training
მნიშვნელოვანი ჰიპერპარამეტრებია: hidden_size, continuous_hidden_size, dropout (ოვერფიტისკენაა ძაან მიდრეკილი). საუკეთესო მოდელი იყო, loss-დ გამოვიყენე, როგორც MAE(), ასევე QuantileLoss(), მედიანის/საშუალოს  არჩევის სტრატეგიით. მედიანის არჩევამ აჯობა.

|**პარამეტრი**|**მნიშვნელობა**|
|---|---|
| hidden_size | 48 |
|continuous_hidden_size|32|
| lr | 1e-3 |
|dropout| 0.3| 
| WMAE| **2014.09** |

ასევე, ვცადე target-ებ-ის ტრანსფორმაცია log1p(y) საშუალებით. აქ გასათვალისწინებელი იყო ისიც, რომ ზოგიერთ კვირაში sale უარყოფითი იყო. მოტივაცია იყო, ის რომ როგორმე დიდი spike-ები დამებალანსებინა და ნორმალიზება მომეხდინა target-ების ისე რომ 9-11 შუალედში ყოფილიყვნენ, შემდეგ კი expm1 საშუალებით უკან დამებრუნებინა. ეს მოახდენდა გრადიენტების ნორმალიზებას, და თეორიულად შეიძლება ტრენინგი უფრო მარტივი გაეხადა. თუმცა ამან wmae გაცილებით უფრო დიდი მოგვცა, ვიდრე სტანდარტულმა მიდგომამ.
